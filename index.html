¿Por qué este ecosistema es único e irreemplazable?

  • Todos los sistemas se integran con SINGA (SQL Server corporativo) — imposible con software genérico del mercado.
  • Los modelos YOLO están entrenados con imágenes propias de la empresa: uniformes, botellas, instalaciones.
  • Los embeddings biométricos (ArcFace) son activos intangibles que pertenecen 100% a la Unidad de Inteligencia.
  • Tres proyectos (Reclut.ai, Contratación Inteligente, Supervisiones) son productos listos para licenciar a terceros.
  • El CRM de Ventas y el módulo de Recursos Humanos de SUMCO ERP se integran directamente con SINGA: pipeline comercial, plantilla y nómina sin re-captura manual de datos.

Por qué esta arquitectura es difícil de replicar

  • SINGA como fuente de verdad única: los 15 sistemas comparten datos en tiempo real. Ningún competidor con SaaS genérico puede replicar esta integración sin reconstruir años de historial.
  • Modelos ML con datos propios: el YOLO conoce los uniformes de la empresa. El ArcFace conoce los rostros. No hay atajos para adquirir esos activos de datos.
  • Stack moderno y homogéneo: Python + FastAPI + PostgreSQL + Docker en toda la capa de servicios reduce fricción y el costo de contratar talento especializado.
  • Cloud-native desde el diseño: no es un monolito migrado — es un ecosistema pensado para escalar en Google Cloud Run desde el primer commit.

Lo que ningún SaaS del mercado puede ofrecer

  • Integración con SINGA: Ningún SAP, Odoo ni HubSpot se conecta nativamente a tu SQL Server. Eso requeriría $30,000–$80,000 USD adicionales solo en integración.
  • Modelos YOLO propios: Entrenados con imágenes de tus uniformes, tus instalaciones, tus operaciones. Ningún proveedor tiene eso — tendría que re-entrenarse desde cero.
  • Embeddings biométricos: Los vectores ArcFace de tus empleados son activos de la empresa. Con cualquier SaaS biométrico, esos datos quedan en servidores de terceros.
  • Datos históricos de SINGA: El caché de 363 MB con años de historial de materiales es un activo que tomó tiempo acumular y no se puede "comprar" a ningún proveedor.

🚧 Áreas de mejora recomendadas
⚠️

No documentado / no confirmado — prioridad para 2026

  • Rate limiting: No hay evidencia de límites de peticiones por IP o usuario en las APIs FastAPI/Flask. Recomendado: slowapi o middleware de Nginx.
  • Cifrado en reposo: PostgreSQL y SQLite no tienen cifrado a nivel de disco documentado. Cloud SQL en GCP sí cifra en reposo por defecto — confirmar en instancias self-hosted.
  • Rotación de credenciales: No hay política documentada de rotación periódica de JWT secrets, API keys de OpenAI/Twilio ni contraseñas de bases de datos.
  • Escaneo de dependencias: No se documenta uso de herramientas como pip-audit, Safety o Dependabot para detectar vulnerabilidades en librerías de terceros.
  • WAF / DDoS: Cloud Run tiene protección básica de Google, pero no hay WAF (Web Application Firewall) explícito para las apps Flask/FastAPI expuestas.
  • Penetration testing: No documentado. Recomendado al menos una vez al año para SUMCO ERP y Reclut.ai antes de abrir a clientes externos.
✅

Postura de seguridad general: Sólida para uso interno, en maduración para exposición externa

  • Los controles de identidad y acceso son completos y bien implementados para un ecosistema de esta escala — JWT + RBAC + bcrypt es el estándar de la industria.
  • La soberanía de datos es un activo real: biometría, historial SINGA y modelos ML nunca salen de la infraestructura propia.
  • El aislamiento en contenedores y la arquitectura cloud-native reducen significativamente la superficie de ataque lateral.
  • Para escalar a clientes externos (Reclut.ai, Contratación Inteligente), implementar rate limiting + escaneo de dependencias es el siguiente paso prioritario.
